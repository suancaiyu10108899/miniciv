# 开发体系反思 (2026-07-13, P1.5 第六个AI)

## 1. 后台进程管理灾难

**问题**: 多次启动的 flatmc 进程互相竞争CPU(两个flatmc-full跑了9.5小时互相拖慢),
数据文件被覆盖/追加/损坏, 大量CPU时间浪费在进程冲突上。

**教训**:
- 启动前必须检查已有进程 (`tasklist | grep`), 杀干净再跑
- 单进程串行 > 多进程并行(避免rayon嵌套竞争)
- RAYON_NUM_THREADS 显式设置, 不依赖默认
- 输出文件加时间戳或唯一后缀, 避免覆盖

**规则**: 任何 `cargo run` 前先 `powershell kill-all`, 然后检查CPU是否归零。

## 2. 流式存储 vs 批量存储

**问题**: 早期工具(flatmc-full)收集所有结果到Vec, 进程被杀后数据全丢。
后期(flatmc-one)每pair立即flush, 中断只丢当前pair。

**教训**:
- 长时间运行的工具必须流式写盘
- `std::io::Write::flush()` 不能省
- 追加模式(`append(true)`)比覆盖模式安全

## 3. 超时陷阱

**问题**: 后台任务有隐式超时, 500 seeds×FlatMC搜索×80T游戏会超时被杀。
数据产出慢但进程还活着→被误判为死循环。

**教训**:
- 长任务用同步执行(run_in_background: false) + 显式长timeout
- 或者拆成独立小任务每个<5分钟

## 4. 探针天花板假说 → 对手模型过拟合

**推演过程**:
第一次发现 FlatMC depth=40→64 AlwaysWhite 从93%跌到63%
→ 假设"探针太弱, FlatMC到平台期了"
→ 自对弈发现 d40>d24(64%)但d40<d32(46%) → 不是平台期, 是策略分化
→ 胜利方式分析: 对AlwaysWhite在d20+出现建设尝试 → 深深度策略变迁
→ 最终确认: 对手模型过拟合(FlatMC优化针对Builder/Rusher但AlwaysWhite不在假设集)

**启示**: 单一指标(胜率)不够, 需要交叉验证(胜利方式/自对弈/不同对手类型)。

## 5. 动作空间分解的正确方式

BC首版失败根因: 硬编码 `decompose_actions` 从Move推断posture→标签混乱。
正确做法: 让FlatMC输出6轴决策时直接记录, 不靠推断。

## 6. 种子数-方差-稳定性的经验关系

| seeds | std(50%WR) | 适用场景 |
|-------|-----------|---------|
| 30 | ±9% | 粗扫(找方向) |
| 50-100 | ±5-7% | 候选筛选 |
| 200-300 | ±3-4% | 矩阵评估 |
| 500 | ±2% | 最终裁决(能区分5pp差异) |

500 seeds的std稳定在0.7-1.5%, 不需要1000。
