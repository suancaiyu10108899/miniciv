## 1. 策略: 全局同步压缩

核心思路: 不是只压经济, 而是**所有轴同步 ×N**。

### 1.1 新参数设计

```
GameConfig 新增:
  tech_turns_mult: f64 = 1.0    // 所有科技研究回合 ×N
  all_tech_cost_mult: f64 = 1.0 // 所有科技(含M/E/C线)成本 ×N
  facility_build_turns: u8 = 1  // 设施建造需要N回合
  unit_cost_mult 生效            // produce_unit + AI can_afford 所有地方

max_turns: 100                   // 阶梯在T100触发(从80→100)
```

### 1.2 目标参数组合(第一轮猜测)

```
max_turns:         100
tech_turns_mult:   3.0    (C1 3T, C3 6T, C4 3T, C5 6-12T → 18-24T链)
all_tech_cost_mult: 3.0   (C线×2.0 + 全局×3.0 = ×6.0? 注意: 和c_line_cost_mult是乘性叠加!)
  → 设计: c_line_cost_mult=1.0(关掉), 只用all_tech_cost_mult=4.0
facility_build_turns: 2   (每设施2回合 → 4设施需8T + 移动时间)
facility_output:     3    (降低产能)
unit_cost_mult:      3.0  (步兵15粮, 骑兵15粮9金)
starting_resources:  10   (开局需要积累)
city_hp:             200  (步兵~45ATK → 5次攻击)
starting_workers:    2
branch_available_turn: 35
```

### 1.3 预期游戏节奏

```
T0-T10:    开局积累期 — 工人建设施, 收集资源, 无法立即研究
T10-T25:   早期扩张 — 开始研究科技, 建设施, 产少量兵
T25-T35:   中期选择 — 红白分叉(约T35)
T35-T55:   路线展开 — 白线爆兵推城 / 红线积累组织度
T55-T80:   终局 — 征服或建设胜利
T80-T100:  阶梯判定(极少触发)
```

### Phase B: 粗扫找方向 (自动, ~30min)

| # | 内容 |
|---|------|
| B1 | 单轴敏感度: 每次只变一个参数, 跑40 seeds Builder vs Rusher, 看对回合数的影响 |
| B2 | 找"游戏长度 > 25T"的参数组合 |
| B3 | 找"建设+征服都 > 15%"的参数组合 |

扫参网格(粗略):
- tech_turns_mult: [2.0, 3.0, 4.0, 5.0]
- all_tech_cost_mult: [2.0, 3.0, 4.0]
- facility_build_turns: [1, 2, 3]
- unit_cost_mult: [2.0, 3.0, 4.0]
- city_hp: [150, 200, 250]
- starting_resources: [5, 8, 10, 12]

约 4×3×3×3×3×4 = 1,296 组合。每组合 40 seeds(Builder vs Rusher) × 2 games = 80局。
→ ~100,000局。release模式 750局/s → ~2分钟。
实际会多很多(因为游戏变长), 估计 5-15分钟。

### Phase C: 细扫找甜点 (自动, ~30min)

在粗扫找到的"游戏>25T且双路径>15%"的区域, 做更高密度扫描:
- 缩小参数范围
- 提高样本量(40→100 seeds)
- 关注阶梯率(< 20%)

### Phase D: 红白分叉点调优

在甜点上, 扫 branch_available_turn [20, 25, 30, 35, 40]:
- 跑 StateAware vs AlwaysWhite, StateAware vs AlwaysRed
- 看 StateAware 全局胜率是否>固定策略
- 看 StateAware 的红/白选择比例是否和"领先/落后"状态一致

**注意**: 分叉点调完后可能需要微调其他参数——这是一个迭代过程。
分叉点影响游戏节奏(什么时候可以选线)→影响红白线的活跃时间→反向影响经济平衡。
预计 2-3 轮迭代。

## 3. 验证策略

### 3.1 每阶段结束检查

- [ ] `cargo test` 全绿
- [ ] AI 是否适配新游戏? (检查所有 ai/*.rs 的 opp 引用和 P1.5 动作使用)
- [ ] 数据文件是否保存? (experiments/v0.10-redwhite/ 下)
- [ ] 文档是否同步? (CLAUDE.md, DECISIONS.md, 记忆)

### 3.2 数据质量检查

- 样本点密度: 如果相邻参数结果跳跃 > 30%, 加密采样
- 平台期检测: 如果连续3个参数结果一致, 扩大范围
- 边界检测: 如果"100%征服"或"100%建设", 说明参数在边界外, 反向调整
- 阶梯率: 任何 pair 阶梯 > 30% → 说明两条路都走不通, 降低难度

### 3.3 FlatMC 裁决有效性

- FlatMC 胜率 > 45% 才能信任它的裁决
- 死机制使用率必须有变化(非零且非恒定)
- 不同 depth 下的表现曲线应有"先升后平"的学习特征

## 4. 风险 + 应对

| 风险 | 概率 | 应对 |
|------|------|------|
| 全局压缩后游戏太慢(>100T全阶梯) | 中 | 降低乘数, 加密扫描 |
| 平衡难找(一条路总碾压另一条) | 中 | 分别调军事/建设侧杠杆, 2-3轮迭代 |
| Agent 在极端参数下行为失当 | 高 | 检查每个AI, 必要时加适应逻辑 |
| FlatMC 仍低于探针 | 中 | 如果分层评估后仍差, 诚实记录,留P2 MCTS |
| NN 训练不收敛 | 中 | 降低网络规模, 延长训练时间 |
| 游戏变慢后矩阵时间不可接受 | 低 | 用更少 seeds 做初筛, 只在甜点区用大样本 |
