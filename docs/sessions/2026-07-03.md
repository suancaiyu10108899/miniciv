## 做了什么

### 全矩阵 6×6 N=500 paired（36,000 局）
- 命令：`python -m prototype.eval_matrix --paired --ais random,greedy,aggressive,flatmc,dqn_trained,evo --games 500 --size 15 --workers 30`
- 耗时：39,844s（~11h），30 workers
- 产出：`experiments/v0.5.0/full-matrix/summary.json`
- 中间保存：每 500 种子 _checkpoint.json（但无法恢复——待改进，Rust 阶段解决）

### 全矩阵关键发现

**Evo 建设胜利统治全场**：
- Evo 平均胜率 75%，打所有 AI 均 72%+
- Evo vs DQN: 72% 胜率，建设胜利率 72%，平均死亡仅 1.4
- Evo vs Greedy: 78% 胜率，建设胜利率 69%
- **42 回合完成 C5，几乎不死兵——其他 AI 在打仗，Evo 在冲科技**

**建设胜利本质是科技胜利**：C5 研究完即获胜，不需要实际建设任何设施。`construction_count()` 统计的是 C1-C5 科技完成数（1-5），不是 facility 数。

**AI 排名**：
| 排名 | AI | 平均胜率 | 特点 |
|------|-----|---------|------|
| 1 | Evo | 75% | 建设胜利统治 |
| 2 | DQN | 74% | 碾压战斗型AI，输Evo |
| 3-4 | FlatMC ≈ Greedy | ~57% | 统计不可分 |
| 5 | Aggressive | 28% | 只赢Random |
| 6 | Random | 19% | 基线 |

### API Key 泄露事件
- **7/1 12:02** miniciv 首次 push 到公开 GitHub（commit `009441b`），API key 在 git 历史中
- **7/2** DeepSeek 后台显示 300M tokens 异常消耗（Pro 200M + Flash 100M）
- **7/3** 排查确认：key 在公开仓库 git 历史中 → GitHub 爬虫扫到后滥用
- **验证**：全矩阵跑 11h 期间零额外 token 消耗 → 实锤泄露，非 CC 后台
- **已处理**：刷新 key、PowerShell profile 更新、.env.example + gitignore .env
- **教训**：API key 绝不进 git 追踪文件；公开仓库 push 前 `git log -S "sk-"`

### C5 成本扫描（多轮调试）
- C5(3,3,3): 82% winrate ← 基线
- C5(6,6,6): 0% winrate ← Evo 旧参数不适应新规则
- **悬崖效应**：成本从 3 到 6 之间无中间地带——太小 Evo 统治，太大 Evo 崩溃
- **根因**：Evo 为固定参数进化，规则一变就废。0% 不证明 (6,6,6) 是坏平衡

### 并行实验基础设施教训
- ProcessPoolExecutor + heredoc 在 Windows 上不兼容
- 模块级 `TECH_TREE_COST` 复制导致内存 patch 无效
- 磁盘编辑 + 多进程竞争 constants.py → 数据污染
- 最终方案：每个 worker 复制 prototype/ 到 temp dir 独立运行
- **教训**：参数扫描脚本应该用独立临时目录或独立 subprocess，避免共享文件
