# UI 开发计划 — 从"看不见"到"看得懂"

> 2026-07-03 | 本文档定义开发工具 UI 的范围、验收标准和阶段划分
>
> 核心原则：**当前 UI 的唯一目标用户是你自己。** 它的作用是让你能看懂游戏里发生了什么——
> 不是为了好看，不是为了给玩家用，是为了加速你自己的设计迭代。

---

## 一、问题诊断

### 当前状态

- `render_ascii.py`：终端 ASCII 渲染。5 种地形 × 5 种兵种 × 设施 × 单位朝向——信息密度远超终端的承载能力。在 15×15 上已经很难阅读，30×30 上基本不可用。
- `render_html.py`：存在但输出静态 HTML，无回合步进、无交互。
- `play.py`：人机对战 CLI。需要输入 `move 0 1 0` 格式的命令，操作开销极高。

### 实际后果

1. **你从未自己玩过自己的游戏。** 不是因为不想，是因为操作门槛太高。
2. **你看不懂 AI 对战。** 36,000 局的数据是抽象的胜率数字，你不知道"为什么 Evo 赢了"——你看不到它的单位移动轨迹、设施建造顺序、战场态势。
3. **设计问题的发现完全依赖批量实验。** facility=8 的困境如果自己有回放工具，看 3 局就能发现——不需要 2,400 局。
4. **设计直觉无法建立。** 策略游戏设计师的核心能力是"看一局回放就知道哪里不对"——这个能力只有通过反复观看才能建立。

### 不是要做什么

- 不是给玩家的游戏 UI（那是 v1.0+ 的事）
- 不是好看的像素美术（参考 Into the Breach 的方向保留到以后）
- 不是完整的游戏客户端（不需要在浏览器里玩——那是平台化阶段的事）

### 是要做什么

**一套"开发工具 UI"——让你能加载回放、步进回合、看清状态。**

---

## 二、设计原则

1. **信息密度 > 美观。** 一个格子上应该同时显示：地形类型（颜色）、己方/敌方单位（字母/图标）、设施（符号）、是否在迷雾中（透明度）。一眼看清。

2. **状态固定可见。** 侧边栏永远显示：回合数、双方资源、科技进度、设施数、存活单位数、胜利类型。不需要"切换面板"。

3. **回合是核心交互单元。** 所有操作围绕回合步进：上一回合 / 下一回合 / 跳到关键回合（有战斗/建设/科技完成的回合）。

4. **零配置。** 加载一个 JSON 文件 → 浏览器打开 → 直接用。不依赖服务器、不依赖数据库、不依赖特定操作系统。

5. **渐进增强。** 先做能用的最小版本，用 5 局回放验证，需要什么加什么。

## 三、阶段划分

### 阶段 1：回放浏览器（当前优先级：P0）

**一句话**：加载一局对局的 JSON → 浏览器里看完整回放。

**验收标准**：

- [ ] 加载 `run_one_game` 产出的 JSON（或 game state 序列），在浏览器中渲染
- [ ] **地图主视图**：
  - 每个格子用颜色区分地形（平原=浅绿 / 森林=深绿 / 山地=灰 / 水域=蓝 / 城市=棕）
  - 己方单位（P0）和敌方单位（P1）用不同颜色边框或底色区分
  - 单位缩写在格子上显示（I=步兵, C=骑兵, A=弓手, S=侦察兵, W=工人）
  - 设施用符号标记（F=农场, L=伐木场, M=矿山）
  - 15×15 的格子足够大，每个格子至少 30×30 px
- [ ] **侧边信息栏**（固定在右侧，不随滚动隐藏）：
  - 当前回合 / 总回合
  - P0：资源（粮/木/金）、科技完成数、construction count、设施数、存活单位数
  - P1：同上
  - 胜利类型（如果有）+ 胜者
  - 所有数字在回合切换时实时更新
- [ ] **回合控制**：
  - ← 上一回合 | 下一回合 →
  - 快捷键：左箭头/右箭头
  - 自动播放模式（1× / 2× / 5× 速度）
- [ ] **关键事件提示**：
  - 如果有 action_log，在发生战斗的回合高亮战斗格子
  - 在科技完成的回合显示"XX 完成了 Y 科技"
  - 在设施建造的回合高亮建造位置
- [ ] 加载一个 100 回合的对局回放，完整浏览时间 < 2 分钟（5× 速度下）

**为什么这是 P0**：这是所有后续 UI 功能的基础。做完这个，你就可以开始看 AI 对战回放了。这是**当前整个项目投资回报率最高的一项工作**。

### 阶段 2：回放增强 + 基础控制（P1）

**一句话**：让回放更好用，并且能自己在浏览器里操作。

**验收标准**：

- [ ] **回合跳转**：下拉菜单或按钮跳转到"关键回合"（定义：有战斗/建设/科技完成/单位死亡的回合）
- [ ] **点击格子查看详情**：点击地图上的格子 → 信息栏显示该格：地形类型、DEF 加成、通行限制、设施归属、单位详情（HP/ATK/DEF/科技加成）
- [ ] **多局对比模式**：同时加载两局回放（同一个 seed 的 P0/P1 互换），并排显示——用于 paired 分析
- [ ] **控制模式**（可选开启）：
  - 选择自己的单位 → 高亮合法移动范围
  - 点击目标格 → 移动
  - 按钮：采集/建造/生产单位/研究科技/结束回合
  - **验收测试**：你自己用这个模式打一局 vs Greedy，操作流畅度让你愿意再打一局
- [ ] 支持 20×20 和 30×30 地图（如果格子变小，用滚动条或缩放适配）

**为什么这是 P1**：阶段 1 让你能"看"，阶段 2 让你能"做"。能自己操作之后，设计直觉的建立速度会快一个数量级。

### 阶段 3：AI 决策可视化（P2）

**一句话**：显示 AI 的"思考"——为什么做了这个决定。

**验收标准**：

- [ ] 在回放中选中一个 AI 控制的单位 → 显示该 AI 在那个时刻考虑的其他选项
- [ ] 对于 Evo：显示每个候选动作的评分（15 个权重 × 候选动作的价值向量）
- [ ] 对于 FlatMC：显示 rollout 统计（"这个动作在 10 次 rollout 中平均得分 3.2，最高 8.0"）
- [ ] **验收测试**：你看一眼 AI 的决策解释，能理解它为什么往那边走

**为什么这是 P2**：这是 VISION.md 里描述的"杀手级功能"的最简原型。它让你能诊断 AI 行为，也让未来给别人看的时候有东西可以展示。

### 不做的事（现阶段）

- 不做玩家注册 / 登录
- 不做服务器端渲染
- 不做 AI 训练面板
- 不做排行榜
- 不做美化（配色方案可以丑——只要信息清楚）
- 不做多语言
- 不做移动端适配

## 四、技术方案

### 推荐方案：纯静态 HTML + JavaScript

**为什么**：
- 零依赖：不需要 Flask/FastAPI/Node.js，不需要数据库
- 跨平台：浏览器在任何系统上都能打开
- 可分享：把 HTML + JSON 打包成一个文件发给别人，别人也能看
- 已有基础：`render_html.py` 已经在产出 HTML

**实现方式**：
- `render_html.py` 升级为 `replay_viewer.py`
- 输出一个自包含的 HTML 文件（JS 内联，不依赖外部 CDN）
- 数据嵌入在 HTML 的 `<script>` 标签里，不需要服务器
- 所有交互逻辑用原生 JavaScript（不需要 React/Vue——你是唯一用户，兼容性不是问题）

### 数据格式

需要一个统一的"回放格式"（不同于当前的 eval 结果 JSON）。建议定义 `GameReplay` 格式：

```json
{
  "version": "1.0",
  "config": { "size": 15, "gen": "balanced", "max_turns": 100, "seed": 42 },
  "turns": [
    {
      "turn": 1,
      "grid": [...],           // 每格的 terrain + facility
      "units": [...],          // 每个单位的 position + hp + type + player
      "cities": [...],         // 城市 HP
      "economies": [...],      // 双方资源
      "techs": [...],          // 双方科技状态
      "events": [...]          // 本回合发生的关键事件
    },
    // ... 每个回合一个 snapshot
  ],
  "result": { "winner": 0, "victory_type": "conquest" }
}
```

这一步需要修改 `game.py` 来产出每回合快照（`snapshot.py` 已有相关功能——可以复用）。

## 六、验收里程碑

### M1：第一次看回放（阶段 1 完成）

**标志**：你加载了 Evo vs Greedy 的一局回放，完整看完了 100 回合，并且能回答以下问题：
- Evo 在第几回合开始建设施？
- Evo 的工人在做什么？（建→采→建？还是纯采？）
- Greedy 有没有试图进攻？如果有，在第几回合？
- 最终谁赢了？怎么赢的？

如果你不能在 5 分钟内回答这些问题 → 回放浏览器不够好用。

### M2：第一次自己玩（阶段 2 完成）

**标志**：你用浏览器控制模式打了一局 vs Greedy，并且：
- 操作流畅（每回合 < 30 秒完成所有单位的行动）
- 你有意识地做了至少一个战略决策（"我应该先建农场还是先研究科技？"）
- 你赢了或者你输了但你知道为什么

如果你打完一局不想再打第二局 → 操作体验不够好。

### M3：第一次理解 AI 行为（阶段 3 完成）

**标志**：你在回放中选中一个 Evo 的单位，看到它考虑的其他选项，并且你至少一次因为看到决策解释而发现了你之前不知道的 AI 行为模式。

---

## 七、总结

**当前 UI 不是"简陋"——是让你看不见自己的游戏。**

一个回放浏览器的 1-2 天投入 → 之后每一轮设计迭代都更快 → 你建立设计直觉的速度快一个数量级 → 到 Rust 阶段时你已经知道游戏"应该是什么样子"而不是"大概是什么样子"。

这是整个 pre-rebuild 阶段**投资回报率最高的一项工作**。
