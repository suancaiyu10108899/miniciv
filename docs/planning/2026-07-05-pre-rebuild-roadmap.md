## 二、六边形决策路径

### 为什么是现在而不是 Rust 重构后

Rust 引擎的坐标系是整个代码库的地基。改坐标系 = 重写 mapgen、movement、render、所有 AI 的编码器、回放格式、replay viewer。在 Rust 里做这个改动的成本是 Python 的 3-5 倍——不是因为 Rust 难写，是因为 Rust 的类型系统让"改个坐标类型"变成传染性重构。

### 阶段 H1：六边形完整原型（1-2 天）

在 `prototype_hex/` 下复制并改写方格引擎的核心模块：

```
prototype_hex/
    __init__.py
    terrain.py         ← 复制（不变）
    unit.py            ← 复制（不变）
    combat.py          ← 复制（不变）
    economy.py         ← 复制（不变）
    tech.py            ← 复制（不变）
    constants.py       ← 复制 + 改坐标相关参数
    mapgen_hex.py      ← 重写：六边形环面地图生成
    movement_hex.py    ← 重写：六方向轴向坐标移动
    game_hex.py        ← 改编：_do_move 方向处理
    ai_greedy_hex.py   ← 适配：方向向量 + 距离计算
    ai_evo_hex.py      ← 同上
    ai_aggressive_hex.py ← 同上
    ai_flatmc_hex.py   ← 同上
    ai_dqn_hex.py      ← 同上
    ai_rulesrandom_hex.py ← 同上
    render_ascii_hex.py ← 重写：六边形终端渲染
    eval_hex.py        ← 适配：距离/地图参数
    eval_matrix_hex.py ← 复制（不变）
```

**不改的文件**（直接 import 方格的）：terrain, unit, combat, economy, tech, constants（部分不变）。

### 阶段 H2：AI 训练 + 调参（2-3 小时）

1. Evo 在六边形上重训（Gen 50, 5min）
2. 快速验证（3 对 AI × 50 seeds）
3. 调 facility/城市参数到六边形上合理的建设/征服分布
4. 中型矩阵（5 AI × 100 seeds, 15min）

### 阶段 H3：方格 vs 六边形对比实验（30 min）

```
对比维度:
    弓手存活率          ← 六边形射程覆盖 18 格 vs 方格 12 格
    骑兵冲锋触发率       ← 六边形 6 方向 vs 方格 4 方向
    征服率              ← 包围/夹击差异
    FlatMC 搜索时间      ← 分支因子 6 vs 4
    Evo 收敛速度         ← 同样 200 代，对比收敛曲线
    P0 偏差             ← 六边形上交替先手是否仍然有效？
```

**验收标准**：至少 3/6 指标六边形优于方格，且无关键指标显著劣于方格 → 选六边形。否则方格够了。

### 阶段 H4：决策（你来做）

我产出对比数据 + 推荐。你操作过 `hex_prototype.py`，看过数据后决定。

## 六、重构开始条件（验收清单）

- [ ] 坐标系已选定（方格 vs 六边形）且有对比实验数据支撑
- [ ] 至少 2 种胜利路径各 > 20%（当前 25.7% 征服 + 32.7% 建设 = 58.4%，初步达标）
- [ ] 无单一 AI 统治（Evo < 65%，DQN < 75%）
- [ ] P0 在所有胜利类型上 ≈ 50%
- [ ] 弓手使用率 > 1.0/game（v0.6.3 mini: 4.4 ✅）
- [ ] 至少一种非 Evo AI 使用骑兵 > 1.0/game
- [ ] 集成测试标准已定义
- [ ] 回放格式 JSON Schema 已定稿
- [ ] 你自己玩了 ≥ 3 局（方格或六边形），确认"这个游戏有趣"
- [ ] 所有文档同步（GAME.md, DECISIONS.md, CC Memory, CLAUDE.md）
