# 方法论讨论归档 — 灵活性、可延展性与子Agent协作

> 2026-07-03 | 本次讨论的核心产出
>
> 问题起点：INFRASTRUCTURE.md 写了"固定字段列表"作为实验标准——但这不够灵活。
> 随着游戏机制增加、新指标被发掘，固定标准会过时。
> 同时，UI 回放浏览器技术难度低但工作量大，适合作为子 Agent 协作的试点。
> 本文档记录达成的共识和设计原则。

---

## 一、核心共识

### 1. 方法论优先于游戏内容迭代

**不是"先把所有东西设计完美再写代码"**（瀑布模型，对 solo 项目不现实）。
**是"在下一次游戏设计迭代之前，确保有工具能看清迭代的结果"**。

具体差距：

| 现在（没有） | 之后（有） |
|------------|----------|
| 改参数 → 跑 3200 局矩阵 → 等 2 小时 | 改参数 → 看 3 局回放 → 10 分钟知道方向 |
| 文档不同步 → 过一周忘参数 | check_docs.py 自动检测 → session 开始就知道 |
| 实验格式每次不同 → 跨实验对比靠手工 | 统一格式 + 自描述 → 脚本自动聚合 |
| Agent 跑飞 → 事后手动清理 | atexit cleanup → 自动停 |

### 2. 灵活性 = 分层 + 自描述 + 渐进式

不定义"必须包含哪些字段"——定义**字段的组织方式**。

- **三层指标**：L1 核心（自动注入，不可跳过）→ L2 标准（默认开启，可关）→ L3 实验性（自由定义，自描述）
- **自描述元数据**：每个实验的 `_fields` 声明每个字段的含义。分析工具读元数据而非硬编码字段名。
- **渐进演化**：当 L3 指标在多个实验中证明价值 → 升级到 L2。L2 被证明普遍有用 → 升级到 L1。

### 3. 子 Agent 做 UI — 一举三得

- **第一得**：产出一个迫切需要的开发工具（回放浏览器）
- **第二得**：验证多 Agent 协作方法论（在真实任务上而非演习）
- **第三得**：建立"主 Agent 设计 + 子 Agent 执行 + 人类验收"的标准流程

UI 是理想试点：技术不确定性极低、验收标准明确、失败代价小、对人的价值高。

## 二、设计原则（归档备查）

### 实验数据格式原则

```
1. 元数据 > 具体字段：不强制字段列表，强制字段的组织和描述方式
2. 实验目的显式声明：question + measurement_method + success_criteria + why_this_method
3. 参数完整快照：不是手写 config.json，是脚本自动从 constants.py 抓取
4. _fields 自描述：分析工具读元数据，不硬编码字段名
5. schema_version：格式会演化，version 保证向后兼容
```

### 子 Agent 任务设计原则

```
1. 任务必须有完成标准（不是"让X变好"，是"X达到Y标准"）
2. 任务必须有预算上限（token数或时间）
3. 技术不确定性必须低（子Agent不负责探索性研究）
4. 产出必须有验收方式（结构化输出 + 可测试的交付物）
5. 失败不能污染核心代码（子Agent工作在隔离环境）
6. 核心文档（VISION/CLAUDE/WORKFLOW/GAME/DECISIONS）人类审核后才能改
```

---

## 三、推荐推进顺序（已讨论确认）

1. **主 Agent 设计**：实验格式规范 + UI 需求文档 + 快照格式升级
2. **人类审核**：确认设计方向
3. **子 Agent 执行 UI**（与步骤 4 并行）
4. **主 Agent 继续做游戏设计反思**：基于 facility=8 结论，思考建设胜利下一步
5. **子 Agent 验收**：人类 + 主 Agent 双重验收
6. **方法论验证完成**：开始下一轮游戏设计迭代，但这次手上有工具
