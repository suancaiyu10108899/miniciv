# 原型开发工作流总结 — 2026-07-02

## 本次会话的实际开发模式

### 单模型全栈 (当前)
- 一个模型读代码 → 分析数据 → 设计修改 → 写代码 → 跑测试 → 写文档
- **优点**: 上下文连贯, 决策一致
- **缺点**: 
  - 主模型上下文被大量代码细节占满
  - 被权限/工具问题打断时恢复成本高
  - 重复性工作(改代码/跑测试)消耗大量token

### Token开销估算

| 操作类型 | 实际token | 理想模式token |
|----------|-----------|---------------|
| 改一个常量 | ~8000 (读文件+python -c+验证+重试) | ~500 (子智能体执行) |
| 跑500局矩阵 | ~2000 | ~500 |
| 数据分析 | ~3000 (必要,无法委托) | ~3000 |
| 写文档 | ~2000 | ~2000 |

## 建议: 主模型 + Flash子智能体模式

```
Opus (主模型/架构师)
  ├ 看数据, 做设计决策
  ├ 规划实验方案
  ├ 审核Flash产出
  └ 写设计文档和分析

Flash × N (执行者)
  ├ "改constants.py第X行, 跑测试"
  ├ "跑500局 X vs Y, 输出胜率/回合/死单位"
  ├ "汇总eval_results下所有JSON"
  └ "写一个XX功能到新文件, 跑测试验证"
```

### 适合委托给Flash的任务
1. **代码修改**: 单文件精准修改 + pytest验证
2. **批量评估**: 跑N局矩阵, 输出表格
3. **数据分析**: 读JSON/CSV, 统计汇总
4. **文档格式化**: 给定数据, 生成markdown表格
5. **速度基准**: bench.py跑一轮, 输出预估

### 不应该委托的任务
1. **游戏设计决策** (需要全局判断)
2. **多文件架构重构** (需要理解模块间依赖)
3. **复杂bug排查** (需要跨文件推理)

## 建议的Skill/工作流

### /eval-matrix
```
参数: --ais random,greedy,aggressive --games 500 --size 15
产出: 胜率矩阵 + 回合分布 + 死单位分布 + 建设/征服/平局比例
```

### /quick-patch
```
参数: --file constants.py --line CITY_HP --value 100
动作: 读取→替换→pytest→输出结果
```

### /analyze-tech
```
动作: 跑N局, 统计每个科技的完成率, 输出分布表
```

### /balance-report
```
参数: --games 1000
动作: 跑全矩阵, 生成平衡报告(先手偏差/胜率/回合/建设率)
产出: markdown文档 + JSON数据
```

## 正式开发(Rust阶段)前的准备

### 开发规范
1. **每次修改后自动跑pytest** (可做成git hook)
2. **数值改动必须跟矩阵数据** (commit message含前后对比)
3. **commit message模板**: `[模块]: 改动摘要\n\n数据: 前后对比\n\nCo-Authored-By: Claude`
4. **大改动前先跑baseline存为JSON** (方便前后对比)

### 理想的每日开发节奏
```
09:00 Opus读昨天数据, 决策今天方向 (~5000 tokens)
09:05 Opus委派3个Flash并行执行 (~2000 tokens)
      Flash1: 改代码+测试
      Flash2: 跑baseline矩阵
      Flash3: 分析昨日数据
09:15 Flash回报结果 (~3000 tokens)
09:20 Opus审核+决策下一步
...循环
```

### 工具链
- pytest: 86测试 < 0.2s, 适合作为每次修改后的快速验证
- eval.py: 单进程批量评估, 适合小规模(≤500局)
- ProcessPoolExecutor: 适合大规模(≥5000局)
- bench.py: 速度基准, 决定矩阵规模
- git: 每次改动独立commit, 方便回退
