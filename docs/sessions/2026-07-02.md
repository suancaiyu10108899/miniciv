#### 深度扫描结果

| depth | rollouts | winrate | P0% | P1% | conquest% |
|-------|----------|---------|-----|-----|-----------|
| 5 | 20 | **90.0%** | 88% | 92% | 40% |
| 10 | 10 | 86.0% | 84% | 88% | 35% |
| 20 | 5 | 76.0% | 80% | 72% | 27% |
| 50 | 2 | 76.0% | 82% | 70% | 36% |

**结论：浅而广 > 深而窄。** 在固定总模拟预算下，多次浅 rollout 明显优于少量深 rollout。depth=5×20rollouts=90% vs depth=50×2rollouts=76%。depth=10→20 是断崖（86%→76%）。


## 决定了什么

### 技术决策
- 文档纪律：CLAUDE.md每session更新 + Session日志 + GAME.md参数真相源
- 中间产物一律塞 session 日志，不建独立文件
- 默认地图尺寸：15×15（30×30 对比实验确认不可行）
- DQN NaN 已修复——根因是梯度爆炸，梯度裁剪 + 降LR 有效
- **FlatMC 深度无效**——最佳配置是 depth=5~10 + 尽可能多 rollout + random policy
- **Python deepcopy 是 FlatMC 的天花板**——greedy rollout 质量高 4% 但慢 8x，不划算
- Hybrid 建议放弃（AI-AUDIT 已标记）

### 流程决策
- **长实验 = 主 Agent 写脚本 → 用户手动跑 → 回来分析。** 不让子 Agent 代跑长任务
- **子 Agent 模型指定 `haiku`**（→ DeepSeek V4 Flash），主 Agent 保留 `opus`（→ V4 Pro）
- **CC 终端不要长时间挂着**——DeepSeek 后台持续消耗 token（推测：实时状态汇报/子 Agent 轮询）
- **并行子 Agent 如果共享代码库** → 必须考虑改动时序依赖，要么串行要么独立分支

### 子 Agent 风控经验
- 文件隔离有效——三个 Agent 修改的文件互不重叠
- 结构化 JSON 输出让验收快速（15s 判定合格/不合格）
- 但子 Agent 不具备平台知识（不知道 Windows 句柄泄漏），复杂实验脚本应由主 Agent 写
- 长时间运行的脚本丢给 Cline 或用户手动跑，不要让子 Agent 在后台干等
