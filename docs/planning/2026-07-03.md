### FlatMC 深度-胜率曲线（random rollout, 400 sims）

```
depth=5, rollouts=20  →  90%  ← 最优
depth=10, rollouts=10 →  86%
depth=20, rollouts=5  →  76%  ← 断崖
depth=50, rollouts=2  →  76%
```

**推论**：在固定总模拟预算下，浅而广 > 深而窄。Python deepcopy 是瓶颈，不是 rollout 质量。

- 并行化后预估：5-10x 加速（取决于 AI 组合）

## 二、P0：FlatMC 参数回滚 + 正式验证

### 目标
将 `ai_flatmc.py` 调到扫描验证过的最优配置，200 局正式评估。

### 具体改动

```python
# ai_flatmc.py — 回滚到最优配置
ROLLOUTS = 10          # 10→5→10（恢复并略增）
ROLLOUT_DEPTH = 20     # 5→20→10（折中：深度和次数平衡）
# Rollout policy: greedy → random（恢复）
```

参数选择理由：
- depth=5×20rollouts=90% 但 20 rollouts 较慢（~68s/局）
- depth=10×10rollouts=86% 且速度更快（~35s/局）
- 最终选 **depth=10 + ROLLOUTS=10 + random**，胜率/速度最优平衡点

### 验证
1. vs Random 200 局 paired，胜率需 > 85%
2. 单局耗时 < 45s
3. vs Greedy 200 局 paired（L2 基线）
4. 如果达标 → 替换当前 ai_flatmc.py 配置

### 产出
- 更新 `ai_flatmc.py` 参数
- L1 + L2 评估数据
- 更新 `docs/AI-AUDIT.md` FlatMC 条目

### 预估耗时
- 200 局 vs Random：~2h（单线程 ~35s/局）
- 200 局 vs Greedy：~2h
- 如果 eval 先并行化 → ~15min

## 四、P2：eval.py 并行化

### 目标
给 `eval.py` 和 `eval_matrix.py` 加多进程并行，利用 32 核。

### 设计

```
单线程（当前）:
  for i in range(n_games):
      r = run_one_game(seed, ai0, ai1, ...)  # 3-130s each

多进程（目标）:
  tasks = [(seed_i, ai0, ai1, ...) for i in range(n_games)]
  with ProcessPoolExecutor(max_workers=16) as pool:
      results = pool.map(_worker_one_game, tasks)
```

### 关键设计决策
- `max_workers`：16（留一半核给系统 + 多个 Agent 并行）
- `chunksize`：动态计算 `max(1, n_games // max_workers // 4)`
- 每个 worker 内部导入 AI 模块（隔离，避免 pickle 问题）
- 进度条：tqdm 或每 10% 打印一次
- 向后兼容：保持 `run_one_game` 单局接口不变

### 验证
- vs Random 100 局：时间对比（单线程 vs 16 workers）
- 加速比需 > 8x（预期 10-12x）
- 结果一致性：同 seed 同 winner

### 产出
- `prototype/eval_parallel.py`（新文件，不影响现有 eval.py）
- 性能对比数据

### 预估耗时
- 开发 + 测试：1-2h（可交给子 Agent）

## 六、P4-P5：暂缓项

### BC 多对手重训
- 前提条件：BC 当前 60% 胜率太低，需要更多样化的训练数据
- 方法：用 Greedy+Aggressive+Random 混合生成行为克隆数据
- 阻塞：需要先生成 ~10,000 局高质量对局数据
- 建议：等 eval 并行化后再做（数据生成会快很多）

### Evo softmax 改进
- 当前 Evo 用硬阈值决策（35% 阈值），随机性过大
- 改进：用 softmax 温度参数替代阈值，平滑决策边界
- 阻塞：优先级低，当前 Evo 75% 胜率已可接受
- 建议：在 FlatMC + DQN 稳定后作为调剂项目
