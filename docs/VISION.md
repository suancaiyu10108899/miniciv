# miniciv — 长远愿景

> 最后更新: 2026-07-03 | 基于两轮原型（MiniCiv AI Lab v7.6 + miniciv v0.5）的经验总结
>
> 本文档是项目的"宪法"。它不描述现在能做到什么——它描述十年后这个项目应该是什么样子。
> 每一个设计决策、每一次技术选型、每一个"做还是不做"的判断，最终都回到这份文档。

---

## 一、这个项目是什么

**一个为 AI 而生的策略游戏平台。AI 不是游戏的附属品——AI 是游戏的存在理由。**

在传统策略游戏里（文明、全战、P 社），AI 是"不得不做"的负担：开发团队投入大量资源做 AI，但大多数玩家不需要好 AI，所以 AI 永远是最低优先级。结果就是靠数值作弊撑场面，玩家抱怨了几十年但没有人有商业动机去解决。

这个项目反过来：**从第一天起，游戏机制的每一个设计决策都以"AI 能不能在这里做出有意义的决策"为第一优先级。** 人类好玩是第二优先级——但这两个优先级不是对立的。一个 AI 真正能玩出深度的游戏，对人类来说也一定比"AI 靠作弊才有挑战"的游戏更有趣。

### 核心差异化

| | 文明六 | 传统 AI 研究平台 | 本项目 |
|---|---|---|---|
| AI 角色 | 对手（障碍），靠作弊 | 研究对象 | **游戏体验的核心** |
| 玩家操作 | 直接控制每个单位 | 不面向玩家 | **配置/训练 AI，观看 AI 执行** |
| 规则设计 | 面向人类直觉 | 简化到无趣 | **面向 AI 决策深度** |
| 数据收集 | 无 | 研究用途 | **游戏对局 = 训练数据，自增长** |
| 竞技性 | PvE 为主 | 论文 benchmark | **PvP（AI 代打）+ 人类可理解** |

## 二、核心信念

### 信念 1：单人做不出 3A 的内容量——但单人可以做出一套复杂到人类玩不了的规则

文明六有 80 个科技、50 个特色单位、30 个奇观、20 个文明——这是靠 60-80 人 × 4 年堆出来的内容量。一个人穷尽一生也追不上。

但文明六的 AI 之所以烂，**恰恰是因为规则太复杂**——行为树 + 脚本的传统方法在 80 个科技 × 50 个单位 × 30 个奇观的组合空间里根本撑不住。

这正是这个项目的切入点：**规则复杂度不是 bug，是 feature。** 一个复杂到人类手写 AI 无法应对、但学习型 AI 可以从中涌现出策略的游戏——这才是单人开发者能做到而 Firaxis 做不到的事。

**复杂度就是你的美术。** 一个 3A 游戏的视觉沉浸感来自美术资源，这个游戏的沉浸感来自系统深度。

### 信念 2：AI 之间对战的观赏性，是这个游戏的核心"内容"

传统游戏的"内容"是关卡、剧情、任务。这个游戏没有这些——它的"内容"是 AI 对战。

一场精彩的 AI 对战应该像一场体育比赛：有战术博弈、有逆转、有风格差异、有"为什么 AI 做了这个决定"的可解释性。回放不是"看看结果"——回放是玩家消费内容的主要形式。

**观赛体验直接决定游戏好不好看。**

### 信念 3：复杂系统 > 复杂内容

国际象棋 6 种棋子 × 64 格 = 几乎无限的策略空间。《Into the Breach》8×8 格子 × 3 种机甲 = 极其丰富的战术决策。

这个项目追求的不是"文明六级别的 80 个科技"，而是"15 个科技之间足够深的交互，以至于选择哪个科技是一个真正有意义的决策"。不是"50 种单位"，而是"5 种单位之间足够精妙的克制关系，以至于阵型选择和地形利用决定了胜负"。

**每增加一个系统，它的价值不在于"又多了一个东西"，而在于"它和已有系统之间产生了多少有意义的交互"。**

### 信念 4：AI-friendly 是一套可操作的设计原则，不是一个口号

说"这个游戏对 AI 友好"很容易。真正做到需要回答具体问题：

- 动作空间设计：是给 AI 高层战略选择（"进攻方向"），还是原子操作（"向左移动一格"）？
- 观察空间设计：给 AI 完整信息还是模拟战争迷雾？特征编码如何让不同地图尺寸下都有意义？
- 奖励设计：sparse reward（赢了才给反馈）还是 dense reward（每一步都有信号）？dense reward 会不会引入 unintended bias？
- 信用分配：一局 100 回合，第 10 回合的一个决策如何影响第 80 回合的结果？AI 能学到这个吗？
- 评估公平性：如何保证"AI A 比 AI B 好"的结论不是随机噪声或者先手偏差？

这些问题没有标准答案，但**必须被持续追问**。每一个游戏机制的设计都需要经过这个镜头的审视。

## 三、玩家分层接入

不是所有人都需要写代码。不同层次的玩家以不同方式参与：

### L1 — 策略观察者（面向最大群体）

- **方式**：选择预制 AI、调整参数滑块（攻击性/防御性/扩张偏好）
- **所见**：AI 性格雷达图 + 决策原因简要解释
- **乐趣**：像足球经理一样配置"战术"，然后看 AI 执行
- **不需要**：写任何代码

### L2 — 规则设计者

- **方式**：手写评分函数，填权重，定义"什么样的局面是好的"
- **所见**：多维评分面板 + 合法动作价值对比
- **乐趣**：自己的策略思想在 AI 对战中验证
- **需要**：基础 Python 知识（或未来提供可视化规则编辑器）

### L3 — AI 训练者

- **方式**：选择训练范式（进化/DQN/PPO/...），调超参数，提交模型权重
- **所见**：完整训练日志 + 实验对比工具 + 排行榜
- **乐趣**：训练出比别人更强的 AI
- **需要**：Python + RL 基础知识

### L4 — 研究者

- **方式**：完整 API 文档 + 标准化 benchmark + 论文模板 + 环境版本快照
- **所见**：统计检验 + LaTeX 表格导出 + 可复现实验配置
- **乐趣**：发表论文、建立学术声誉
- **需要**：学术研究能力

## 四、AI 决策可视化——这个项目的"杀手级功能"

大多数策略游戏的 AI 是黑箱——玩家看到 AI 移动了一个单位，不知道为什么。在这个项目里，AI 的"思考过程"本身是游戏体验的一部分：

```
┌─────────────────────────────────┐
│  🗡️ 步兵 (3,4)                  │
│  ─────────────────────────────  │
│  选择的行动: 进入森林格          │
│                                 │
│  为什么？                       │
│  ✅ 森林 DEF+10（防御加成）      │
│  ✅ 离敌方城市距离缩短 2 格     │
│  ❌ 放弃了平原上的冲锋路线      │
│                                 │
│  AI 性格:                       │
│  攻击性 ████████░░ 80%         │
│  防守   ██████░░░░ 60%         │
│  扩张   ████░░░░░░ 40%         │
└─────────────────────────────────┘
```

这不是锦上添花的 UI——这是**让 L1 玩家理解"AI 在做什么"的唯一方式**。没有它，AI 对战就是一堆单位在地图上乱动。有了它，每一局都是一场有叙事张力的比赛。

## 五、长远游戏设计方向

### 地图

- **当前**：方格 + 四邻 + 环面。简单、清晰、实现成本低
- **未来考虑**：六边形（6 方向 > 4 方向，移动和距离计算更自然）。但这是重大决策——改写所有底层系统。应在 v1.0 之前用原型验证，不要拖到 Rust 重构之后
- **拓扑**：环面（消除角落安全）是确定的。这是"AI-friendly"的一个具体例子——任何位置拓扑等价，AI 不需要学习"角落是安全的"这种特殊规则

### 科技

- **方向**：不追求数量（文明六 80+ 科技），追求交互质量
- **模式**：DAG 而非纯线性。互斥选择（M2 vs M3）创造真正的路线分化。当前 13 节点就是这种思路的雏形
- **长期**：让不同科技路线产生足够不同的游戏风格——走军事线的 AI 和走经济线的 AI 玩起来像两个不同的文明

### 经济

- **核心张力**：建 vs 产 vs 兵。"工人应该建新设施还是从已有设施采集？应该投资经济还是暴兵？"——这些 trade-off 是策略游戏的引擎
- **方向**：保持工人-设施模型的简单性，但在上层的资源转换链条上增加深度（比如：粮→养更多工人→更多设施→更多产出→支撑更大军队）

### 战斗

- **方向**：少而精的兵种，清晰的克制关系，地形作为"战力倍乘器"而非单纯的加减
- **固定伤害 vs 随机伤害**：当前固定伤害对 AI 训练极度友好（精确计算、清晰信用分配），但可能让战斗缺乏戏剧性。需要一个不牺牲 AI 友好度的"可控随机"方案

### 胜利条件

- **核心设计**：至少保证一条非零和路径（建设/科技/经济），让"不打架也能赢"成为可能
- **方向**：多条路径之间的博弈——"对方在冲科技胜利，我是应该跟冲还是进攻打断他？"——这种跨路径的战略决策是策略游戏最深层的乐趣

## 六、技术演进路线（方向性，不定日期）

### 当前阶段：原型探索（Python）

**定位**：快速验证游戏机制和 AI 方法。不追求性能，追求迭代速度。

**什么时候这个阶段算完成？**
- 有至少一个版本，你自己坐下来玩了 30 分钟觉得有趣
- 至少两种不同的获胜路径在人类对战中都有可行性
- 全矩阵 P0 偏差 ≤55%，AI 排名稳定可复现
- 已经确定"六边形要不要做"、"地图默认多大"、"核心兵种是哪些"——这些地基决策不再变了

### 中期阶段：引擎固化（Rust/C++）

**定位**：Python deepcopy 是搜索型 AI 的天花板。当游戏设计基本稳定后，用系统语言重写核心引擎。

**性能目标**：
- 100k 局/秒 Random vs Random（单核）
- 1k 局/秒 FlatMC（单核）
- 亚毫秒级 GameState 深拷贝
- 每局内存 < 1KB

**关键设计**：
- GameState Copy-on-Write（支持高效 tree search）
- 确定性 RNG（seedable，保证可复现）
- 统一 AI trait（所有 AI 可互换）
- Python bindings（PyO3，训练脚本继续用 Python）

**这不是现在要做的事。** 在游戏设计稳定之前写 Rust 是在移动目标上做工程。但可以现在就为 Rust 做准备——记录哪些 Python 模式在 Rust 中有更好的等价物，哪些设计决策会影响迁移难度。

### 长期阶段：平台化

**定位**：从"我一个人的项目"变成"社区可以参与的平台"。

- 社区 AI 提交 + 自动评估 + 排行榜
- 赛季机制（规则更新 → 旧 AI 分区 → 新赛季开始）
- AI 竞赛 + 学术 benchmark 正式发布
- 回放分享 + 观赛体验
- 如果社区规模够大：Steam EA / 赞助 / 企业合作

## 七、美术与音效策略

**原则**：风格统一 > 技术复杂度。2D 优先（前 5 年不考虑 3D）。不追求"好看"，追求"有辨识度"。

**分阶段**：

| 阶段 | 美术 | 音效 |
|------|------|------|
| Python 原型 | ASCII + 简单 HTML 彩色格子 | 无 |
| Rust 核心 | 2D tile（16×16 像素）+ 2-3 套调色板 | 移动/战斗/胜利各 0.5s 8-bit |
| 平台化 | AI 辅助统一风格迭代 | 按需扩展 |

**参考**：《Into the Breach》的极简像素风格——信息清晰、风格统一、开发成本可控。

## 八、世界观

当规则复杂度达到需要叙事框架来包裹时再展开。由项目作者规划核心大纲，AI 辅助资料查阅和扩展。

**时机**：不是现在。等游戏机制足够有趣之后，世界观才有东西可以包裹。

## 九、商业化思路（远期，不做承诺）

| 模式 | 说明 | 时机 |
|------|------|------|
| Steam EA 买断制 | 核心玩法免费 + 外观/AI 皮肤付费 | 有可玩版本 + 社区反馈后 |
| 学术 benchmark | 免费排名，企业版 API 付费 | 有多篇论文引用后 |
| 云算力赞助 | 厂商提供训练算力换品牌曝光 | 有稳定用户群后 |
| 赛季赞助 | 企业赞助赛季命名权 | 有足够玩家规模后 |

**但商业化不是目标。** 如果项目永远只是"我自己的 AI 学习平台"，那也完全值得做。不设 DDL，不设盈利压力。

---

## 十、不做的事（同样重要）

- **不做 3D**：前 5 年不考虑。2D 能表达的信息密度已经足够
- **不做多人实时对战**：回合制 + AI 代打 = 天然异步。实时多人需要完全不同的技术栈和设计哲学
- **不追"像文明六"**：追它做不到的方向，而不是复制它已经做到的方向
- **不做移动端**：除非有专门团队。桌面端（Terminal / Web）足够
- **不着急发论文**：等有站得住脚的结论再发。差的论文比不发更糟糕
- **不设 DDL**：这是一个长期爱好，不是产品交付

## 十一、容易被忽视但重要的工程问题

| 问题 | 说明 | 何时需要 |
|------|------|---------|
| **AI 版本管理** | 环境更新后旧 AI 战绩需分开统计。需要环境版本快照 + 向后兼容 | 有社区 AI 提交后 |
| **实验可复现性** | 随机种子 + 环境版本 + 参数快照。已在 prototype 阶段铺垫 | 现在就开始 |
| **反作弊** | 如果未来有社区 AI 提交，需要沙箱限制网络访问 | 平台化阶段 |
| **社区冷启动** | 种子 AI 阶梯——新玩家前 10 局打种子 AI，不需要等待匹配 | 有排行榜后 |
| **数据集版本化** | 游戏规则变化 → 旧对局数据不再有效。需要版本标签 | 有训练数据集后 |
| **文档退化** | 代码和文档不同步是 solo 项目最常见的腐化模式 | 一直在发生，持续对抗 |

---

## 十二、最后的话

这个项目不是为了"做一个像文明六的游戏"——是为了探索一个文明六永远无法成为的方向：一个从第一天起就为 AI 竞赛而设计的策略游戏平台。

在文明六里，AI 是游戏的附属品。在这个项目里，AI 是游戏的存在理由。

单人做不出 3A 的 UI 和美术。但单人可以做出一套复杂到人类玩不了的规则——规则的设计不依赖美术资源。复杂度就是你的美术。

最终这个项目能不能走到平台化、有没有人用、赚不赚钱——都不重要。重要的是它是我理解"如何设计复杂系统"的载体。从这里学到的东西会远超这个项目本身。

不设 DDL，不给自己压力，哪怕最后只是自己的 AI 学习项目也好。先做下去。
