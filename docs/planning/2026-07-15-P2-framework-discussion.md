## 一、已覆盖的议题（本轮讨论成果）

### 1.1 游戏设计：骨架 vs 血肉

**核心判断**：用户的三点理由（框架先定避免重构、深度来自复合作用、需要能外发的完整游戏）方向正确，但执行上需要修正——不是"一起全做完再调"，而是"一起冻结 Schema，按波次激活，每波消融验证"。

**已识别的骨架层**（必须 W0 冻结 Schema）：

| # | 骨架组件 | 涉及的数据结构 | 依赖它的子系统 |
|---|---------|--------------|-------------|
| 1 | 地图与空间模型 | `Tile` 扩展：settlement_id、resource、river、road、visible_to、last_seen | 聚落、FOW、战略资源、河流、道路/后勤 |
| 2 | 聚落实体 | `Settlement`（替代当前 `City`）：tier、specialization、tiles、population、food_surplus、integration、buildings | 行政力、供应链、Flexible Zoning、征服消化、专业化 |
| 3 | 资源与经济模型 | `Economy` 扩展：五货币+三战略资源+容量型资源+天命+政策状态+依赖度+失真度 | 双树、贸易、红白深化、核威慑、事件系统 |
| 4 | 可见性系统 | `VisibilityLayer`：visible_tiles、last_seen_terrain、last_seen_units | FOW、AI 接口、侦察兵价值、卫星/无人机 |
| 5 | 双树系统 | 科技节点（科技点解锁）+ 制度节点（文教/天命解锁）+ 互斥选择 | 诸子百家、时代过渡、路线选择、红白分叉 |

**已识别的血肉层**（Schema 上留坑，实现按波次）：
行政力（先用供应链距离衰减替代）、后勤、战争烈度、贸易依赖度、超级工程、核威慑、灾害、红白深化（信息失真等）、事件系统

**建议推迟到 P3 的**：
能源转型体系、完整海洋开发、地图剧本多样化、完整多人外交（合纵连横/朝贡体系）

### 1.2 开发体系：波次推进

**建议的波次划分**（与来源 C 版本略有调整）：

| 波 | 内容 | 验收标准 |
|---|------|---------|
| **W0** | Schema 冻结 + 清理（46 clippy / 7 bug / 文档腐烂/ 废弃 bin） | 新 GameState 可初始化；旧测试全绿；GAME.md 重写 |
| **W1** | 25×25 + FOW + 多聚落骨架 | 3 聚落/玩家可跑 100T+ 不 panic；侦察兵被至少一个 AI 使用 |
| **W2** | 经济体系（五货币+战略资源+铜双铸+供应链） | 消融矩阵：每个新机制单独开关时指标显著移动 |
| **W3** | 双树 + 百家 + 时代过渡 | 5 学派 EGTA 无 >60% 支配；时代过渡不产生 >20% 前期定胜负 |
| **W4** | 红白落地 + 战争烈度 + 行政力 | R-1/R-3 人机双验证（P1.5 回归套件） |
| **W5** | WASM UI + 复盘 + alpha 发布 | 真人完局率 >50%；人类版 R-1 出现苗头 |

**每波结束强制回归**：R-1 处境选线、R-3 频率依赖、先手 ≤55%、FlatMC 深度无平台期 ——任何波杀死了 P1.5 验证过的信号，该波机制回炉。

### 1.3 文档体系重组

当前 CLAUDE.md 已膨胀至 200+ 行，承载了交接文档+进度表+导航+铁律+历史裁决的混合职责。建议拆分为：

```
CLAUDE.md           — 精简到 ~80 行（当前阶段 + 快速启动 + 关键约束 + 指向其他文档）
docs/P2-PLAN.md     — 合同（范围/边界/波次/验收/止损）
docs/SCHEMA.md      — 冻结的数据结构定义（P2 唯一真相源）
docs/DESIGN.md      — 设计决策（延续 DECISIONS.md，按系统分节）
docs/EXPERIMENTS.md — 实验注册表（参数/数据路径/结论）
docs/ARCHIVE/       — 过期文档归档
```

### 1.4 四个可能的误判

1. **"先都做进去再调数值"在 P2 规模上行不通**：20 个子系统的参数空间是百万级——必须依赖消融实验（机制独立开关）而非全参数扫描
2. **社区发布目标需要校准**：50-100 个硬核策略玩家的反馈 > "Steam 10 万份"。核心重玩性来自 红白×百家×地形×对手 的组合爆炸，不是子系统数量
3. **FOW 对所有 AI 的重构工作量被低估**：手写 AI 的评估逻辑全部假设全信息，改 `decide` 签名为 `PlayerView` 后，它们的行为会退化
4. **行政力是最危险的新系统**：太强→扩张不划算（违反 4X 根本乐趣），太弱→死机制。建议 W1 先用供应链距离衰减+征服整合度替代，不够再加

## 二、本轮新提出的议题（待讨论）

### 2.1 UI 架构：P2 需要什么样的界面？

**现状**：
- `replay_viewer.html`（106 行）：单文件 HTML，用 Canvas 渲染六边棋盘，支持回合滑动+播放。纯前端，读本地 JSON 文件
- 无游戏内 UI——当前引擎是纯 headless 的，只有 CLI 工具
- 无人类玩家交互界面

**P2 需要的最少 UI**：
1. **游戏界面**：人类能在棋盘上看到 FOW 过滤后的地图、自己的单位/聚落、已知的敌方信息，能发出 Action（移动/建造/研究/外交等）
2. **复盘界面**：比当前 `replay_viewer.html` 更完整——不仅看单位移动，还能看资源曲线、科技树进度、支持度/天命变化、关键事件
3. **实验数据浏览界面**：矩阵结果、消融对比、单局深度回放——让开发者（你）不用翻 JSON 就能判断"这个机制起作用了吗"

**技术选型问题**：
- 纯 WASM + Canvas（延续当前 `replay_viewer.html` 的思路，但升级到完整应用）
- 还是 WASM + Web 框架（React/Vue/Leptos）？
- 数据怎么传给前端？JSON 文件？本地 HTTP API？WebSocket？

**关键约束**：你是单人开发者。UI 容易吃时间黑洞——"把一个按钮调好看"可以花三天。P2 的 UI 标准应该是"功能完整，不丑即可"——中国史皮肤是 W5 的事。

### 2.2 数据基础设施：P2 的数据量会爆炸

**现状**：
- P1.5 总实验数据：110MB（含 120 万局对局、164K BC 样本）
- 单局回放 JSON：28KB（1v1 简单配置）到 7.2MB（2v2 完整回放）
- 数据格式：JSON 文本文件 + CSV
- 数据索引：无——靠 CLAUDE.md 手写列表和文件名命名约定
- FlatMC 深度曲线：28 个独立 CSV 文件（d2~d96），每个 1KB
- 总文件数：experiments/v0.10-redwhite/ 下 73 个文件

**P2 的数据量预估**（25×25 棋盘 + 多聚落 + FOW + 200-300T）：

| 数据类型 | P1.5 规模 | P2 预估规模 | 增长因子 |
|---------|----------|-----------|---------|
| 单局回放（全帧） | 28KB-7MB | 500KB-50MB | ~10×（更多格、更多单位、更多回合） |
| 单局终态快照 | ~1KB | ~5KB | 更多玩家状态字段 |
| 矩阵（6AI×500seeds） | ~28KB JSON | ~500KB-5MB | 取决于 instrumentation 深度 |
| BC 自对弈数据 | 164K 行 CSV (23MB) | 500K-2M 行 | 更大棋盘需要更多样本 |
| 消融矩阵 | 无（P1.5 没做） | N 机制 × M seeds × 2 状态 | 完全新的数据类别 |

**核心问题**：
1. **JSON 行存撑不住**：单局回放 50MB JSON 时，人类打开浏览器会卡。需要二进制格式或分帧存储
2. **没有查询能力**："FlatMC d24 在所有 Builder 对手上的征服率是多少？"——当前需要翻几十个 CSV/JSON 文件
3. **数据散落**：实验参数、结果、回放之间的关系靠文件命名和大脑记忆

### 2.3 数据管理方案（需要设计）

**分层存储策略**：

```
L0: 实验注册表 (docs/EXPERIMENTS.md 或 SQLite)
    — 每次运行的参数、AI列表、seeds、数据路径、结论摘要
    — 人类和 AI 都读这个来找"上次那个实验在哪"

L1: 聚合指标 (SQLite 或 Parquet)
    — 每局的摘要行：seed, AI组合, 胜者, 胜利方式, 回合数, 各玩家终态字段
    — 支持 SQL 查询："SELECT avg(turns) FROM games WHERE ai_p0='FlatMC' AND victory='Conquest'"
    — P1.5 数据量：120万行 × ~20列 = ~50MB → SQLite 轻松处理

L2: 完整回放 (二进制分帧格式，如 custom binary 或 gzipped JSON chunks)
    — 只在需要深看时加载
    — 平时不碰

L3: 原始训练数据 (CSV/Parquet，如 BC 样本)
    — 训练脚本直接读
```

**建议**：W0 就引入 SQLite（或 DuckDB），建一张 `games` 表和一张 `experiments` 表。所有 bin 工具统一写入这个库。CLI 查询替代翻 CSV。

### 2.4 Instrumentation 策略：不要每次重新发明要测什么

**当前问题**：`PlayerEndState`（eval.rs:24-37）是 P1.5 手工定义的 13 个字段。如果 P2 加了 20 个新系统，每次评估都要重新决定"我要记录什么"——然后发现上次忘了记某个关键字段，整批实验重跑。

**建议：Universal Instrumentation Profile**

每个系统在 Schema 冻结时同时定义自己的 instrumentation 字段：

```rust
trait Instrument {
    /// 返回"本系统在当前对局中的关键指标"的键值对
    fn instrument(&self, gs: &GameState, pid: u8) -> HashMap<String, f64>;
}
```

- 粮食供应链：`food_produced`, `food_consumed`, `supply_route_length`, `supply_break_turns`
- 行政力：`admin_cap`, `admin_used`, `tributary_count`, `integration_avg`
- 红白：`branch_chosen`, `crisis_count`, `org_redeemed`, `distortion_level`
- ……

**所有 bin 工具自动收集所有已注册的 instrumentation 字段。** 不需要每次实验重写收集逻辑。实验者只需要在 SQL 查询里选感兴趣的列。

### 2.5 AI 管理：探针、手写、训练、策略分类

**现状**：
- 17 个 AI agent（含 BC），分布在 7 个源文件中
- 命名混乱：`probes.rs` 里塞了 9 个探针 + 3 个复杂 agent（StateAware/AlwaysWhite/Adaptive）
- 无 AI 注册表——找"有哪些 AI"需要翻 `ai/mod.rs` + `bin/replay.rs`（后者有手工 AI 列表）
- BC 的权重文件是独立的 JSON，Evo 的权重也是 JSON——没有统一的"模型文件"格式

**P2 的 AI 数量预估**：

| 类别 | 数量 | 说明 |
|------|------|------|
| 探针 AI（固定策略） | 25-40 | 每个新系统至少+1个探针（供应链探针/贸易探针/工程时机探针...） |
| 手写启发式 AI | 5-8 | Greedy/StateAware/FlatMC 等，随机制复杂化需要升级 |
| 训练型 AI | 3-5 | Evo/NN/BC v2（turn-plan）/ 可能的 DQN 实验 |
| 5 学派特化 | 5+ | 法家爆发/道家防守/墨家守城/兵家信息/儒家外交——EGTA 策略池基础 |
| 总计 | **40-60** | 管理方式必须升级 |

**需要设计的东西**：

1. **AI 注册表**：一个集中位置声明"有哪些 AI、各是什么类型、依赖什么数据"
   ```rust
   // 不要在 bin/replay.rs 里手工 match 了
   AgentRegistry::list_all()  // → Vec<AgentMeta>
   AgentRegistry::create("FlatMcAgent(d=24)")  // → Box<dyn Agent>
   ```

2. **AI 分类体系**：
   - **Probe**：固定策略，无状态，用作"机制是否死"的试纸
   - **Heuristic**：手写分层决策，有状态（Greedy/StateAware）
   - **Search**：在线搜索（FlatMC 及未来 MCTS 变体）
   - **Learned**：离线训练权重（Evo/NN/BC），需要加载模型文件

3. **统一的模型文件格式**：替代当前 Evo JSON / BC JSON 各自为政
   ```json
   { "model_type": "EvoLinear", "version": 2, "weights": {...}, "training_config": {...} }
   ```

4. **AI 版本管理**：同一个 AI 的不同版本（Evo V1/V2/V3、Greedy 调参前后）如何命名和追溯？建议：`agent_name/version` 格式，version = git commit hash 前 7 位 + 训练日期

### 2.6 前后端分离：需要吗？

**当前架构**：纯 Rust 单体——引擎 + AI + CLI 工具全在一个 crate 里。`replay_viewer.html` 是一个独立的前端文件，手动加载 JSON。

**P2 的三个使用场景**：

| 场景 | 需要什么 |
|------|---------|
| **批量实验**（你 90% 的时间） | Rust CLI/库，高性能，不需要 UI |
| **人类试玩** | 浏览器 UI，WASM 引擎 + 前端渲染 |
| **复盘分析** | 浏览器 UI，读回放数据文件 |

**建议：不需要传统意义上的"前后端分离"（Rust 服务器 + JS 客户端通过 HTTP 通信）——你不需要服务器。**

正确的架构是 **三模式**：

1. **Headless 模式**（已有）：`cargo run --bin exp -- ...`——纯 Rust，跑实验
2. **WASM 库模式**：`miniciv-core` 编译为 WASM，暴露 `init_game_wasm()` / `step_game_wasm()` / `get_state_json()` 等函数
3. **Web 前端**：静态 HTML/JS（或简单框架），import WASM 模块，渲染和交互

前端和后端（WASM 引擎）跑在同一个浏览器进程里——没有网络请求，没有服务器，没有数据库运维。数据存储用浏览器 LocalStorage/IndexedDB。

**唯一可能需要"服务器"的是**：人类对局数据的社区收集（alpha 发布后）。那时可以加一个最简单的上传端点——但这是 W5+ 的事。

### 2.7 回放系统升级

**当前 `snapshot.rs` 的问题**：
- 每帧全量序列化整个 GameState（所有格、所有单位、所有经济数据）→ 单帧体积大
- `TurnSnapshot` 的字段是 P1.5 的——没有支持度、组织度、天命、聚落信息等 P2 字段
- `MAP_W`/`MAP_H` 硬编码——改棋盘尺寸后回放渲染会错
- `events: Vec::new()` ——事件系统完全没接入回放

**P2 需要的回放格式**：

```
GameReplayV2 {
    config: ReplayConfigV2,         // 一次性元数据（地图、AI、参数）
    initial_state: StateSnapshot,   // Turn 0 的完整快照
    turns: Vec<DeltaSnapshot>,      // 每回合的增量（只记录变化）
    result: ReplayResult,
}
```

增量快照的优势：25×25=625 格，假设每回合平均只有 20 个格有变化（单位移动/设施建造/战斗），增量格式比全量小 30×。

**回放前端升级**：
- 资源曲线（折线图：粮食/产能/财富/科技/天命随时间变化）
- 科技树进度（每回合解锁了什么）
- 事件时间线（危机/灾害/兑换/宣战——水平滚动条）
- 单位统计（产了多少步兵/骑兵/弓手，死了多少）
- 支持 P2 新渲染：聚落边界（Flexible Zoning 的多格着色）、FOW 迷雾层、道路/河流水域格

### 2.8 人类对战集成

**人类玩家如何和 AI 对战**：

1. 浏览器 UI 展示 FOW 过滤后的棋盘（`PlayerView`）
2. 人类点击单位 → 显示合法移动范围（高亮可达格）→ 点击目标格 → 生成 `Action::Move`
3. 点击城市/聚落 → 显示可生产/可建造/可研究选项
4. 点击"结束回合" → `step_game` → AI 回合（可能瞬间完成，不需要动画）
5. 循环

**关键设计问题**：
- 人类玩家是否需要等 AI "思考"？FlatMC d24 的搜索可能需要几百毫秒——对批量实验无所谓，对人类玩家是明显的卡顿。解决方案：AI 计算在 Web Worker 里异步跑，UI 显示"对手思考中..."
- 操作错误怎么办？撤销？——棋类游戏的撤销容易（无随机），你的游戏有确定性 RNG——撤销需要保存 RNG 状态，可以实现但需要设计
- 信息展示：人类需要看到什么？FOW 下"已知但过时的信息"怎么在 UI 上区分（当前可见 / 曾经可见现在不可见 / 从未可见）？

### 2.9 可能遗漏的其他问题

**🔴 高优先级**：

1. **地图生成器的公平性验证**：`paired` 设计假设同一 seed 的两局地图相同。但如果 P2 引入不对称资源分布（你多铁他多马），需要确保 paired seed 下双方出生区的"总价值"可比较——否则 paired 设计的前提动摇。需要一个 `evaluate_starting_position()` 打分函数来验证。

2. **Config 参数爆炸**：当前 `GameConfig` 已有 40+ 字段。P2 再加 20 个系统的开关和参数 → 60-80 字段。需要分组（`MapConfig` / `EconomyConfig` / `CombatConfig` / `FOWConfig`...）否则 init 代码不可读。

3. **`rem_euclid` 的环面包装在 25×25 上的行为**：当前 15×15 上环面包装导致"往左走绕到右边"的距离是 15。25×25 上这个距离是 25——但城际距离 8-12 格时，环面包装变得更罕见。需要验证环面是否仍然有意义。

4. **团队系统的 N>4 泛化**：当前 `teams: Vec<u8>` 支持任意分队，但胜利判定（建设/征服/阶梯）的团队聚合逻辑在 >4 玩家时有没有边界 bug？需要 N=6/8 的测试。

**🟡 中优先级**：

5. **WASM 编译的确定性**：当前引擎的确定性依赖 `ChaCha12Rng` 的跨平台一致性——这在 WASM 上是否成立？需要在 W0 就做一个 WASM 编译 + 同 seed 对比测试。

6. **并行评估的 I/O 瓶颈**：P1.5 数据 110MB、单机 SSD 没问题。P2 单局回放 50MB × 1000 seeds = 50GB——需要决定"完整回放只保存多少比例"（如每对 AI 只保存 10 局完整回放，其余只保存 L1 摘要）。

7. **中文/英文双语**：代码注释和 doc comment 当前是中文。如果 alpha 发布到国际社区——需要决定是先全中文（发布时再翻译）还是从现在开始就双语。建议：代码标识符保持英文（Rust 惯例），注释可以中文，但公开 API 的 doc comment 从现在开始补英文。

**🟢 低优先级（P3+）**：

8. **声音**：alpha 不需要。但 4X 游戏的声音设计（时代切换音效、战斗音效、危机警报音）对沉浸感贡献大——先记着，P3 再说。
9. **Mod/脚本支持**：如果社区要加新兵种/新科技/新事件——需要什么接口？目前不设计，但 Schema 设计时留一点扩展性（如 `TechNode` 用配置文件而非硬编码 `static ALL_NODES`）。

## 四、第二轮讨论：地图尺度与军事模型（2026-07-15 追加）

### 4.1 地图尺寸修正

基于"格子是子功能单元 + 城市 20-40 格 + 多聚落 + 玩家间距"的估算，25×25 不够：

| 玩家数 | 推荐尺寸 | 格数 | 每人格数 | 已开发利用率 | 城际行军（步兵移4） |
|--------|---------|------|---------|-----------|------------------|
| 2 | **35×35** | 1,225 | 612 | ~40% | 10-18 回合 |
| 4 | **45×45** | 2,025 | 506 | ~49% | 12-22 回合 |
| 6 | **50×50** | 2,500 | 417 | ~60% | 15-25 回合 |

50×50 = 2,500 格。Canvas 渲染无压力，A* 路径规划 < 1ms。没有技术天花板。

### 4.2 视野科技线

视野不应是固定值——应是一条独立的科技成长线：

| 来源 | 视野 | 时代 | 条件/代价 |
|------|------|------|----------|
| 裸眼（无单位） | 1（仅邻格） | 远古 | — |
| 侦察兵 | 4 | 远古 | 基础单位 |
| 骑兵斥候 | 6 | 封建 | 需马 |
| 瞭望塔（设施） | +3 固定点 | 封建 | 工人建造 |
| 烽燧（沿驿道） | 瞭望塔间信息共享 | 封建 | 驿道科技 |
| 热气球 | 8，持续 3T | 封建末 | 一次性消耗 |
| 侦察机 | 直线 15 格窄带 | 现代 | 需油 |
| 无人机 | 10 + 站岗 | 现代 | 军民两用 |
| 卫星（一星） | 全图 60%/10T | 未来 | 需稀土 |
| 卫星链 | 全图持续 | 未来 | 超越胜利前置 |

### 4.3 兵种升级链（四时代）

```
远古（4）：民兵/斥候/弓手/战车（需马）
封建（5）：步兵/骑兵（需马）/弩手/攻城器（需铁）/工程兵
现代（5）：步枪兵/装甲（需油）/炮兵/防空炮/工程营
未来（4）：机械化步兵/无人机群/导弹部队（需稀土）/核弹（特殊）
```

关键原则：**每代兵种功能差异化，不是换皮加数值。** 弓手→弩手→炮兵→导弹部队：射程 2→3→4→8，但越来越贵、越来越需要资源。战车→骑兵→装甲→机械化步兵：移动链上每代解决不同问题。

移动力基准（35×35 尺度）：步兵移速 4-5，骑兵 6-7。道路移动消耗减半。离开补给范围后移速递减。

### 4.4 城市防御：三层模型

城市 20-40 格，"1 格 1 单位"无法防守。新模型：

**第一层：外围（农田/牧场/作坊区）**
- 敌方自由进入掠夺（有即时资源收益）
- 不触发民兵抵抗
- 守方面临"出城保护 vs 龟缩保城"抉择

**第二层：城市本体（坊市/官署/工坊区）**
- 敌方进入消耗 2 倍移动力
- 每格每回合受民兵伤害（基于城市人口 × 天命系数）
- 有驻军的格 → 民兵伤害 ×2
- 城墙节点 → 敌方进入相邻格时被远程攻击一次

**第三层：核心区（宫殿/市政厅，1-3 格）**
- 敌方步兵需占领核心区持续 N 回合才陷落
- N = 村子 1T / 城镇 2T / 城市 3T / 专业化 4T
- 占领期间守方可以夺回

### 4.5 驻军槽位

| 聚落等级 | 战斗单位驻军槽 | 特殊 |
|---------|-------------|------|
| 村子 | 1 | — |
| 城镇 | 3 | 驻军可部署在边界任 3 格 |
| 城市 | 5 | 含 1 个"军营区"可额外堆叠 2 单位 |
| 专业化城市 | 6 | 按专业类型获特殊驻军效果 |

### 4.6 军团作为科技解锁（非默认机制）

- 封建"方阵"→ 2 步兵合并为步兵方阵（占 2 连续格，正面 DEF ×1.5，移速 -1）
- 现代"装甲师"→ 3 格编队（装甲+机械化步兵+炮兵）
- 非默认——路径规划复杂度在科技解锁后才存在

### 4.7 城市管理：区域涂抹而非逐格微操

玩家视角：地图上属于某城市的连续格子用颜色标记。玩家点"工业区"工具 → 拖动画笔覆盖若干格 → 这些格变成工业区。引擎自动在区域内填放最优设施组合。城市距离限制（BFS 距离 ≤ 聚落等级环数）防长条城市。

### 4.8 玩家角色：皇帝模式 + 将军模式

**核心洞察**：如果游戏规则复杂到人类微操不过来，玩家的角色可以从"每格每单位操作"变为"设定战略 + 选择 AI + 监控进展 + 关键干预"——和项目的原始愿景"AI 是游戏的存在理由"完全一致。

四种玩家深度：
- 纯观战：选 AI、看
- 战略层：设定方向、任命 AI、少而重要的决策
- 混合层：关键战斗自己打，其余委托
- 微操层：传统 4X 操作一切

**AI 架构：分层子 AI**
```
战略 AI（选目标、分资源）→ RL/BC
  ├─ 城市管理 AI（区域布局）→ 可独立训练
  ├─ 军事 AI（战场机动）   → 可独立训练
  ├─ 外交 AI（贸易/联盟）   → 规则型 + RL
  └─ 侦察 AI（探索方向）   → 规则型
```

分层训练效率远高于端到端——每层有独立奖励函数。
