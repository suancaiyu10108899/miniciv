# 开发基础设施 — 待解决的问题与标准

> 2026-07-03 | 基于两轮原型中暴露的系统性问题
>
> 本文档回答一个问题：**除了游戏设计本身，还有哪些东西需要在"认真开发"之前想清楚？**
> 每一项都包含：当前状态 → 问题描述 → 解决标准（什么算"够了"）

---

## 一、总览：问题地图

```
游戏本体之外的问题分为五个领域：

  A. 安全与进程管理    — 防止自毁（API泄露、死循环、磁盘占满）
  B. 实验标准          — 保证数据可信（可复现、可对比、不浪费算力）
  C. 文档体系          — 保证知识不丢失（设计意图、已知bug、踩过的坑）
  D. AI/Agent 协作纪律 — 保证工具为人服务而非反之
  E. 未解决的方法论问题 — 我们至今不知道怎么衡量的事
```

---

## 二、A 域：安全与进程管理

### A1. API Key 安全

**当前状态**：已处理。旧 key 已删，`.env.example` + `.gitignore .env` 已建立。但防御是被动的——它阻止了同一个错误再发生，但没有阻止"类似的新错误"。

**残余风险**：
- 训练脚本可能把 key 硬编码进某个 `train_*.py`，且这个文件不在 gitignore 里
- 第三方工具（SB3、wandb 等）可能从环境变量读取 key 并上传到外部服务
- 未来如果加 CI/CD，key 可能通过 CI 日志泄露

**解决标准**：
- [ ] 写一个 `check_leaks.py` 脚本：扫描所有追踪文件中的 `sk-`、`Bearer`、`api_key` 等模式——不是给人跑，是给 pre-commit hook 跑
- [ ] 建立 `.git/hooks/pre-commit`：自动运行 leak check
- [ ] 任何新引入外部服务的依赖（wandb、huggingface hub 等），使用前必须先确认它不会静默上传数据

**判定"够了"**：当 leak check 脚本存在且绑定到 pre-commit，且所有外部依赖的数据上传行为已被确认。

### A2. 孤儿进程

**已发生**：多次。ProcessPoolExecutor 崩溃后 worker 残留、CC 终端挂着消耗 token、实验脚本 Ctrl+C 后 Python 子进程继续跑。

**残余风险**：
- 残留的 Python 进程占用 CPU/内存，影响后续实验性能
- 在笔记本上跑大规模实验时，残留进程可能导致过热或电池耗尽
- 多个残留进程竞争 constants.py 导致数据污染（已发生过）

**解决标准**：
- [ ] 每个实验脚本开头加一个 `atexit.register(cleanup)` ——确保无论什么方式退出，子进程被回收
- [ ] 建立 session 启动/关闭的标准检查：session 开始时 `check_processes`，结束时 `cleanup_processes`
- [ ] WORKFLOW.md 中已有的清理命令（`Get-Process python* | Stop-Process`）应自动化而非靠人记得

**判定"够了"**：当连续 5 个 session 没有出现过残留进程。

### A3. 磁盘空间

**已发生**：旧项目 data/ 目录 1.1GB，531MB trajectory JSON。当前项目 `experiments/` 5.8MB——暂时安全。

**残余风险**：
- 大规模训练产出的模型权重（`.pt`、`.npz`、`.zip`）可能快速占满磁盘
- 实验原始数据（每局 JSON ~2-5KB）在全矩阵规模下快速增长
- Windows 笔记本磁盘空间通常比 Linux 服务器紧张

**解决标准**：
- [ ] 定义"保留什么、删什么"的规则并写进 WORKFLOW.md：
  - 模型权重：只保留 top-3 checkpoint（按验证胜率），其余自动删除
  - 实验原始数据：保留 summary.json + 每对的前 10 局原始数据（用于 debug），其余原始 JSON 可删
  - 训练日志（JSONL）：保留最近 3 次训练
- [ ] 写一个 `cleanup_experiments.py` 脚本实现以上规则

**判定"够了"**：当实验目录不再手动清理（脚本自动维护），且磁盘使用量在预期范围内。

## 三、B 域：实验标准

### B1. 什么中间数据必须保留？

**当前状态**：checkpoint 机制已实现（eval_matrix.py 每 500 seeds 写 `_checkpoint.json`），但没有定义"哪些指标是必须记录的"。

**问题**：当你事后想分析"为什么 Evo 在 seed=237 那局赢了"，你只有胜率数字，没有那局的状态序列。而且不同实验记录的字段不一致——有的有经济指标，有的没有。

**解决标准**：
- [ ] 定义实验产出的三级数据：

| 级别 | 内容 | 何时保留 |
|------|------|---------|
| L1 摘要 | 胜率 + 胜利类型分布 + 平均回合 + 置信区间 | **永远保留** |
| L2 指标 | 每局的 P0/P1 经济、科技、设施、死伤 | 保留（压缩后 ~100 bytes/局） |
| L3 回放 | 每回合完整 GameState 序列 | 仅保留重要的/代表性的 seed（如"Evo 唯一一次建设胜利"）|

- [ ] 在 `eval_matrix.py` 和任何新的实验脚本中，统一输出格式包含 L1+L2
- [ ] L2 指标列表标准化：`winner, victory_type, turns, p0_facilities, p1_facilities, p0_construction, p1_construction, p0_alive, p1_alive, p0_dead, p1_dead, p0_resources_total, p1_resources_total`

**判定"够了"**：当两个不同实验产出的 summary.json 有完全相同的字段结构，可以直接 diff。

### B2. 实验可复现性

**当前状态**：部分实现。seeded RNG、config.json 记录参数。但以下不可复现：
- 多进程下的任务调度顺序（不影响结果但影响中间输出顺序）
- 操作系统级随机性（Windows ProcessPoolExecutor 行为）

**解决标准**：
- [ ] 任何实验的 config.json 必须包含：git commit hash、Python 版本、命令行参数、完整参数快照（不只是改动的那一个参数）
- [ ] 写一个 `reproduce.py` 工具：给定一个 config.json，能重新跑出相同结果（在统计误差范围内）
- [ ] 关键实验（全矩阵、参数扫描）必须在至少两个不同时间跑过，结果统计不可区分

**判定"够了"**：当你换了一台电脑，根据 config.json 能复现出 95% CI 范围内的结果。

### B3. 统计报告标准

**当前状态**：eval_matrix 输出 summary.json——有胜率、stddev、95% CI。但缺少：
- 效应量（Cohen's d 或类似）——统计显著不代表实际显著
- 多重比较校正（全矩阵 6×6=15 对比较，假阳性风险）
- 胜率差异的经济显著性（"AI A 比 AI B 多赢 3%，但这个差异在 500 局里 p<0.05"→ 这个差异到底重不重要？）

**解决标准**：
- [ ] summary 至少包含：winrate + 95% CI + Cohen's d（vs 基线对手）
- [ ] 全矩阵报告包含：排名 + 每个排名的置信区间 + 统计不可区分的组（如"FlatMC 和 Greedy 统计不可区分"）
- [ ] 定义"实际显著"的最小效应量：胜率差异 < 5% 且 p < 0.05 → 统计显著但实际不显著，不在报告中强调

**判定"够了"**：当你读一份实验报告时，能区分"A 真的比 B 强"和"A 比 B 好一点但在噪声范围内"。

## 四、C 域：文档体系

### C1. 文档防退化

**已发生**：constants.py DEFAULT_SIZE=30 vs GAME.md 默认=15、GAME.md 不同步 facility 前置条件、AI-AUDIT 排名基于旧规则数据。

**根因**：文档和代码之间的同步依赖人的记忆。人一定会忘。

**解决标准**：
- [ ] 在 `constants.py` 顶部加注释：列出哪些 `.md` 文件引用了这里的值（作为同步 checklist）
- [ ] 写一个 `check_docs.py` 脚本：自动检测 constants.py 和 GAME.md 之间的不一致
  - 解析 constants.py 中的数值定义
  - 解析 GAME.md 中的表格
  - 报告不匹配
- [ ] Session 结束的 checklist 中加一项："run check_docs.py"

**判定"够了"**：当 check_docs.py 存在且每次 session 结束都跑。

### C2. Bug/Insight Journal

**当前状态**：旧项目有（7 bugs + 4 insights），新项目没有。

**为什么重要**：旧项目 B01（一行引用 bug 废掉 5 次 RL 训练）如果在两年后重现，你不会记得是同一件事——除非有记录。这是 solo 开发者最容易被时间击败的地方。

**解决标准**：
- [ ] 在 `docs/journal/` 下建：
  - `BUGS.md`：格式 = 日期 + 现象 + 根因 + 复现条件 + 修复方案 + 预防措施
  - `INSIGHTS.md`：格式 = 日期 + 现象 + 解释 + 对设计的启示
- [ ] 不需要迁移旧项目的内容（那是旧游戏的经验），但从现在开始记录
- [ ] 每个 session 结束时检查：有没有新 bug 或新 insight 值得记录？

**判定"够了"**：当你踩到一个坑，第一反应是"这应该写进 BUGS.md"，然后真的写了。

### C3. 实验文档

**当前状态**：实验目录有 config.json，但质量参差不齐。部分实验没有描述"为什么要跑这个实验"。

**解决标准**：
- [ ] 每个实验目录包含一个 `README.md`（可以只有 5 行）：
  - 假设（"我们预期 facility=8 会让 Evo 胜率降到 50%"）
  - 方法（多少局、什么对手、什么参数）
  - 结论（"实际上 Evo 胜率 75%，建设率 0%"）
  - 对设计的启示（"facility=8 太高，需要降低或换机制"）
- [ ] experiment config.json 模板标准化（已在 WORKFLOW.md 中定义，但需要强制执行）

**判定"够了"**：当三个月后再看任何实验目录，不需要翻代码就能理解它做了什么、为什么、结论是什么。

## 五、D 域：AI/Agent 协作纪律

### D1. Agent 行为边界

**已发生**：Agent C 后台超时后 Python 进程残留、并行 Agent 共享代码库导致隐式依赖、Agent 输出自由文本主 Agent 难以验证。

**解决标准**：
- [ ] Agent 任务规范模板（已在 WORKFLOW.md 中定义）需要强制执行——不接受模糊任务描述
- [ ] Agent 代码改动必须有对应的测试改动（至少 smoke test）
- [ ] Agent 产出的文档改动必须经过主 Agent（你的人类判断）审核后才能合并
- [ ] 禁止 Agent 修改 WORKFLOW.md、CLAUDE.md、VISION.md——这些是人类的"护栏"，不能让 Agent 自己改护栏

**判定"够了"**：当连续 5 次 Agent 任务不需要因为"Agent 做错了什么"而回滚。

### D2. Token 预算管理

**已发生**：CC 终端长时间挂着消耗 300M tokens。子 Agent 超时跑死循环灌垃圾。

**解决标准**：
- [ ] 每次使用 Agent 前明确预算上限（token 数或时间）
- [ ] Agent 任务必须有完成标准（不是"让 X 变好"，是"X 达到 Y 标准，Z 次验证通过"）
- [ ] 长任务（>30min）不交给 Agent 后台跑——写脚本，人手动执行
- [ ] Session 结束强制关闭 CC 终端（不是"挂着明天继续"）

**判定"够了"**：当不再出现"Agent 跑了很久但不知道它在干什么"的情况。

### D3. 文档修改权限

**已发生**：Agent 修改文档后未同步更新相关文档，导致不一致。

**解决标准**：
- [ ] 核心文档（CLAUDE.md、GAME.md、DECISIONS.md、WORKFLOW.md、VISION.md）的修改必须有明确的"同步影响分析"——改了一个地方，还有哪里需要改？
- [ ] 文档修改的 commit 必须标注"同步检查：已更新 X, Y, Z"或"无需同步"
- [ ] 所有 Agent 产出的文档修改，人类必须 review 后才能 commit

**判定"够了"**：当连续 3 个 session 没有发现"改了一个文档但忘了改另一个"的情况。

## 六、E 域：未解决的方法论问题

这些问题不是"需要修好的 bug"，而是"我们至今不知道怎么回答的问题"。它们比代码缺陷更深层——它们是研究空白。

### E1. 侦察兵价值怎么衡量？

**为什么重要**：5 种单位中，侦察兵是唯一"不直接参与战斗也不直接参与经济"的单位。它的价值全在于信息。但信息价值是所有游戏机制中最难量化的。

**可能的衡量方法**：
- **消融实验**：跑同样的 AI 矩阵，一次有侦察兵（视野 3），一次侦察兵视野=0（等效于没侦察兵但还占一个单位格）。胜率差异 = 侦察兵的信息价值。
- **行为分析**：对比有/无侦察兵的 AI 对局中，单位的移动路径是否有显著差异。如果有——侦察兵的信息改变了战术决策——那它有价值。如果没有——AI 根本不用侦察信息——那侦察兵就是废物。
- **地图尺寸交互**：在 15×15 vs 30×30 上分别测侦察兵价值。预期：大地图上信息更稀缺 → 侦察兵更有用。

**解决标准**：
- [ ] 跑至少一个消融实验（有侦察兵 vs 无侦察兵）
- [ ] 结论写进 AI-AUDIT 或新开一个 insight
- [ ] 如果侦察兵在所有条件下都无价值 → 考虑移除或重新设计（比如让侦察兵有"标记目标使友军 ATK+5"的能力）

**判定"够了"**：当你能用数据回答"侦察兵在 15×15 上让 AI 胜率变化了 X%，在 30×30 上变化了 Y%"。

### E2. 迷雾在 AI 对战中有没有意义？

**为什么重要**：迷雾是游戏机制中计算成本最高的之一（每回合更新每个单位的视野）。如果 AI 对战不在乎迷雾（因为 AI 的行为不受信息不完整影响），那它就是纯粹的性能浪费。

**当前**：迷雾代码写了（`fow.py`），但所有 AI 都没有被限制在迷雾范围内做决策——它们实际上能看到全图。

**解决标准**：
- [ ] 实现"AI 只能看到视野内的单位"的约束
- [ ] 跑矩阵对比：全信息 vs 迷雾限制，AI 的胜率有没有变化？
- [ ] 如果无变化 → 迷雾对 AI 对战无意义 → 标记为"仅人类游戏模式需要"
- [ ] 如果有变化 → 分析哪些 AI 受影响更大（预期：搜索型 AI 受影响最大，因为搜索质量依赖信息质量）

**判定"够了"**：当 DECISIONS.md 里有一条关于"迷雾在 AI 模式下的设计决策"的条目。

### E3. 地图尺寸和策略多样性的关系

**假设**：更大的地图 → 更多空间做经济 → 建设胜利更强。但 30×30 上战斗型 AI 接触不到对方 → 战斗型 AI 废了。

**我们不知道的**：
- 多大的地图是"甜点"——既给经济策略留空间，又不至于让战斗策略无意义？
- 建设胜利的可行性是否和地图尺寸正相关？如果是，也许不同尺寸需要不同的 facility 门槛？
- 是否有第三种策略（"扩张型"——不 rush 科技也不 rush 战斗，而是铺大量设施靠经济碾压）在中等尺寸地图上会出现？

**解决标准**：
- [ ] 至少跑 3 种尺寸（15, 20, 25）× 至少 3 种 AI 的交叉对照实验
- [ ] 结论写进 DECISIONS.md（不一定现在做，但要在"参数锁定"之前做）

**判定"够了"**：当 DEFAULT_SIZE 的选择有实验数据支撑，而不是"15 是当前的默认因为没时间测别的"。

### E4. AI 行为的"可解释性"怎么衡量？

**为什么重要**：VISION.md 里写了 AI 决策可视化是杀手级功能。但如果 AI 的决策本质上是不可解释的（比如神经网络黑箱），这个功能就做不了。

**我们不知道的**：
- 当前哪种 AI 的决策最适合可视化解释？（Evo 的 15 个权重可能是最容易的——"因为 Weight[attack]=0.8 > 阈值"）
- 用户（L1 玩家）能接受什么级别的解释？"AI 选了 A 因为评分函数说是最好的" 够不够？还是需要更深层的因果解释？

**解决标准**：
- [ ] 至少对一种 AI（建议 Evo）实现一个最简可用的决策解释输出：每个单位，输出它考虑的前 3 个选项和每个选项的评分
- [ ] 人类测试：你看着这个解释，能不能理解 AI 为什么做了这个决定？
- [ ] 如果不理解 → 解释需要怎么改进？

**判定"够了"**：当你给一个不懂代码的人看决策解释，他能大概理解"AI 为什么往那边走"。

### E5. 学习型 AI 的"参数鲁棒性"怎么保证？

**已发生**：Evo 在 C5(3,3,3) 下胜率 82%，在 C5(6,6,6) 下 0%。参数一变，训练好的 AI 就废了。

**我们需要的**：
- 要么：游戏参数锁定后，在固定参数上训练 AI（旧方法）
- 要么：训练时就随机化参数，让 AI 学会适应参数变化（参数条件化训练，见 planning/2026-07-03.md）
- 要么：使用搜索型 AI（FlatMC）因为在线搜索天然适应规则变化

**这是个战略选择，不是技术问题。** 它决定了：
- 如果选"锁定参数再训练"：游戏设计必须完全稳定后才能投入 AI 训练
- 如果选"参数条件化训练"：训练成本更高但 AI 对参数变化鲁棒
- 如果选"搜索型 AI 为主"：Rust 的优先级大幅提高（Python 太慢）

**解决标准**：
- [ ] DECISIONS.md 中有一条关于"AI 训练策略 vs 参数锁定时机"的明确决策
- [ ] 这个决策影响了后续的工程优先级

**判定"够了"**：当你不需要在每次改游戏参数时纠结"要不要重训 AI"。

## 七、优先级排序

不是所有问题都需要现在解决。按"如果不解决会直接导致项目崩掉"排序：

### P0 — 不解决会自毁

| 问题 | 状态 | 下一步 |
|------|------|--------|
| A1. API key 安全 | ⚠️ 已修复但无预防 | 写 pre-commit leak check |
| A2. 孤儿进程 | ⚠️ 反复发生 | atexit cleanup + session 自动化 |

### P1 — 不解决会浪费大量时间

| 问题 | 状态 | 下一步 |
|------|------|--------|
| B1. 实验数据标准化 | ❌ 未做 | 定义标准字段模板 |
| C1. 文档防退化 | ❌ 未做 | 写 check_docs.py |
| D1. Agent 行为边界 | ⚠️ 已定义但未强制执行 | 严格执行 |
| D2. Token 预算 | ⚠️ 已吸取教训 | 持续遵守 |

### P2 — 不解决会不知道自己在干什么
