# P1.5 收尾总结 (2026-07-13, 第六个AI)

## 开发体系反思

### 做对了的
- 每次数据产出立即 git commit+push, 不攒到最后
- 规划文档写清楚才写代码
- 流式写盘(flatmc-one 每pair flush), 避免批量存储丢数据
- 三条Evo升级路并行探索, 快速试错

### 反复踩的坑
1. **进程管理**: 多次启动flatmc进程互相竞争, 两个进程跑了9.5h互相拖慢
   → 规则: 每次启动前 `powershell kill-all`, 验证CPU归零
2. **后台超时**: 反复在同一个坑跌倒——后台任务有隐式超时, 长进程被杀
   → 规则: 长任务用同步执行+显式长timeout
3. **CLAUDE.md未及时更新**: 方向改变时交接块过时
   → 规则: 每个阶段性结果产出后立即更新CLAUDE.md
4. **路径问题**: cargo run从miniciv-core/执行, 相对路径指向错误位置
   → 规则: 所有输出路径用 `../experiments/` 前缀

### 方法论文收获
- 探针天花板≠搜索平台期(需要自对弈+交叉验证来区分)
- 线性模型87%天花板是架构瓶颈不是数据瓶颈
- 500 seeds的std稳定在0.7-1.5%, 够区分5pp差异
- 胜利方式分析(征服vs建设比例)能揭示策略变迁, 单看胜率不够
